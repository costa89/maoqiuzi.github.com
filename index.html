毛球子好为人师

去Github上看看

引子：Git的传说（The Git Parable）

我计算机方面的知识大部分都是自学的，经常要自己摸索。在摸索的过程中，我走了很多弯路，有了问题不知道到哪里去寻找答案。我常常想，互联网上大家分享了那么多的知识，为什么我还是在走弯路呢：是我的方法有问题？还是我的水平有问题？

想要知道到底是哪里出了问题，不妨回忆一下：一般地，在我想要自学某个新知识的时候，非常痛苦，往往上网看了一通下来，知识的结构还是很松散。我发现，网上的资料要么是高度的概括，要么是非常细枝的细节，要么属于”误打误撞“类型的知识：看过之后只能即时的解决一些问题，并不能提供任何启发。

我想，可能并不是我的方法或者水平有问题，因为有时候，在偶然的情况下，我会遇到一些非常好的材料，会有一种茅塞顿开的感觉。所以，我现在感觉，应该这些资料并没有好好的针对初学者，就算当我回过头来再看这些材料的时候，也总会有一些高屋建瓴的感觉。

实际上，当我学一项新知识的时候，我真正希望的顺序应该是：先把握住整体，然后根据需要，有选择地深入到某一个细节中去。所以，我希望能找到这样一种入门教程：先给我一个简单的总体介绍，让我梳理出一个脉络；然后根据不同知识的复杂度不同，给出各种继续深入学习的方法。

这种模式我认为是最好的，恰好前一段时间我在学习Github的时候，看到了这么一篇文章，原名叫做《The Git Parable》，作者没有像一般的介绍github的文章那样，开始先介绍几个命令，演示一番Github的用法，然后：Tadaa~~~！实际上，作者完全没有介绍任何Github的命令！然而，我却觉得这比我所见到过的任何入门性Github的介绍都要好。看完这篇文章，我才理解了我为什么需要在Github上敲那些命令，它除了引入了麻烦和复杂之外，能带给我什么好处？而解决了这些困惑之后，我再沿着它给的链接去学习具体的Github应用，也就是那众多的命令，我发现我不再是机械地背诵那些命令，而是我当有了需要，主动去寻找那些命令，然后欣然地发现Github的设计者们早就替我想好了，放了一个比我的需求还要完美的工具在那里，拿来用就好了。

所以花了几天时间,把它翻译过来，希望能够有更多的人知道这样一篇文章；有更多的人了解Github；更希望能从这篇文章中得到一些启迪，总结出某种规律，让大家能够更好地分享知识。

人之患，在好为人师

我们的老祖先说过：”人之患，在好为人师“，我本人呢，还真就是好为人师，就是水平还不够罢了。但我不觉得这是”人之患“。自忖自己尚有那么有一点儿天赋，能把复杂的事情简单的讲给别人，我发现，为了达到交流的目的，很多时候必须啰嗦，要在开讲之前把大家带到同一个背景知识框架里来。尽管如此，过往的反响也都还不错！所以我想发展一下这个长处，总结出某种规范性的东西，翻译一些、原创一些技术方面的资料，同南来北往的朋友们一起交流探讨。我真心地希望看到大家都能好为人师，少一些功利，用平和的心态，把各自的经验分享出来，互相交流促进，提高我们整体的水平。

尽管水平有限，我个人还是很顽固的喜欢规范化，所以希望和各位看官一起达成某种共识，最好能建立某种规范。尽管现阶段看来超出了我的能力范围，但是我想一步步向着这个方向努力，让我们能用一种更规范化的方式分享知识。
